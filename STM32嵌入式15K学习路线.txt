【1. 当前岗位要求画像】
======================

我按 2026 年 6 月能查到的岗位信息和官方技术资料做了归纳：

· 15K 左右的 STM32 嵌入式岗位，普遍不再只要求会点亮 LED、会写串口，而是要求能独立负责产品软件模块。
· 15K-30K 岗位常见关键词：STM32、GD32、ESP32、ARM Cortex-M、C/C++、FreeRTOS、RT-Thread、lwIP、UART、SPI、I2C、ADC、PWM、CAN、RS485、USB、Ethernet、MQTT、Modbus、LoRa、Bootloader、OTA、LCD、LVGL、TouchGFX、示波器、逻辑分析仪。

· BOSS直聘页面能看到 STM32 相关岗位中出现 15K-30K、15K-20K 等职位，部分要求 1-5 年经验、本科或大专以上，并强调 STM32/ESP32、驱动、RTOS、协作修复 BUG 等能力。
· 远程工作者 2026-04-28 的嵌入式岗位明确提到全职 15K+ 起，要求 C/C++、STM32/ESP32、UART/I2C/SPI/ADC/PWM/GPIO、FreeRTOS、MQTT/CoAP、示波器/逻辑分析仪、文档和现场调试。
· 智联招聘来源的 15K-30K 岗位提到 STM32/GD32/AGM32、USB/UART/SPI/I2C/RS232/RS485、uCOS/FreeRTOS/RT-Thread、MQTT/LoRa、测试、故障分析、技术文档、自动化测试等。

【2. 15K 岗位能力分层】
==========================

2.1 入门可就业：8K-12K
--------------------------------

能做：

· 使用 STM32CubeMX / STM32CubeIDE / Keil 建工程。
· 会 GPIO、EXTI、TIM、PWM、ADC、UART、I2C、SPI。
· 会 HAL 库调用，能看懂基础原理图。
· 能用串口打印调试，能使用 ST-Link 下载和单步调试。
· 能写简单传感器驱动、按键、LED、蜂鸣器、OLED 显示。

短板：

· 遇到 DMA、中断优先级、RTOS 时序、协议异常、硬件噪声、现场偶发死机时会比较吃力。

2.2 15K 主战能力：12K-18K
----------------------------------------

必须能做：

· 独立完成外设驱动、通信协议、状态机、环形缓冲区、日志系统。
· 熟练使用 FreeRTOS：任务、优先级、队列、信号量、互斥锁、事件组、软件定时器、任务通知。
· 能把 UART/I2C/SPI/CAN/RS485/ETH 放进 RTOS 架构，不靠 while 轮询硬撑。
· 能完成 Modbus RTU、CAN 自定义协议、MQTT/HTTP 基础接入。
· 会 lwIP 的 DHCP、静态 IP、TCP、UDP、Socket/Netconn/Raw API 基本使用。
· 会看原理图，能用示波器和逻辑分析仪定位电平、时序、波形、通信问题。
· 能写设计文档、接口文档、测试报告。

2.3 20K+ 拉开差距能力
------------------------------

重点是系统化：

· FreeRTOS 任务架构设计：优先级、栈、堆、阻塞、超时、资源保护、死锁规避。
· lwIP 稳定性：MAC/PHY、DMA 描述符、pbuf、内存池、重连、吞吐、丢包、长连接、看门狗。
· 屏幕驱动和 GUI 性能：LTDC、FMC/8080、SPI 屏、RGB565、帧缓冲、DMA2D、双缓冲、触摸、LVGL/TouchGFX。
· Bootloader / IAP / OTA：Flash 分区、向量表跳转、CRC、版本管理、断电保护、回滚。
· 故障定位：HardFault 分析、栈溢出、内存越界、Cache 一致性、DMA 异常、I2C 总线挂死、CAN 总线错误。
· 自动化测试：Python 串口测试、协议回归测试、硬件在环测试、日志分析。
· 量产意识：配置保存、校准、出厂测试、参数版本、异常恢复、升级失败恢复、文档交付。

【3. 必学技术清单】
==================

3.1 C语言与嵌入式C
------------------------

· 指针、数组、结构体、联合体、枚举、函数指针、回调函数。
· const、static、volatile、extern、inline 的真实使用场景。
· 位运算、寄存器操作、掩码、位域的优缺点。
· 内存对齐、大小端、结构体打包、通信帧解析。
· 栈、堆、全局区、静态区、Flash、RAM 的关系。
· 环形缓冲区、链表、队列、查表法、状态机、事件分发。
· 错误码设计、断言、日志等级、模块化头文件。
· C 语言常见坑：野指针、数组越界、重入问题、中断与主循环共享变量、宏副作用。
· 基础 C++：类、构造析构、RAII、虚函数成本、模板慎用、extern "C"。STM32 岗位不一定要求重 C++，但 TouchGFX 会涉及 C++。

3.2 ARM Cortex-M 与 STM32 基础
------------------------------------------------------

· Cortex-M0/M3/M4/M7 差异，FPU、DSP、Cache、MPU。
· 启动文件、复位流程、向量表、中断入口、SystemInit()、main()。
· NVIC、中断优先级、抢占优先级、子优先级。
· SysTick、HAL Tick、FreeRTOS Tick 的关系。
· 时钟树：HSE、HSI、PLL、AHB、APB、外设时钟。
· GPIO 模式：输入、输出、复用、模拟、上拉、下拉、开漏、推挽。
· DMA：普通模式、循环模式、半传输、传输完成、中断配合。
· Flash：擦除粒度、写入限制、扇区/页、读保护、Option Bytes。
· I-Cache / D-Cache：尤其 STM32F7/H7 上 Ethernet、DMA、LCD 常见 Cache 一致性问题。

3.3 STM32 常用外设
----------------------------

· GPIO/EXTI：按键消抖、外部中断、低功耗唤醒。
· TIM：基本定时、PWM、输入捕获、编码器模式、输出比较。
· ADC：单通道、多通道扫描、DMA采样、校准、滤波、基准电压。
· DAC：波形输出、模拟量测试。
· UART/USART：阻塞、中断、DMA、空闲中断、环形缓冲区、命令行。
· I2C：EEPROM、温湿度传感器、总线扫描、ACK/NACK、超时恢复。
· SPI：OLED/TFT、Flash、全双工/半双工、片选、DMA。
· CAN/FDCAN：过滤器、标准帧/扩展帧、错误状态、总线关闭恢复。
· RS485：方向控制、半双工、Modbus RTU、终端电阻。
· USB：CDC 虚拟串口、MSC、HID 基础。
· Ethernet：MAC、PHY、RMII/MII、DMA描述符、lwIP 网络接口。
· RTC：时间保持、备份寄存器、低功耗唤醒。
· IWDG/WWDG：独立看门狗、窗口看门狗、喂狗策略。
· SDIO/SDMMC + FatFS：日志、数据记录、文件系统异常恢复。
· FMC/FSMC：8080 并口屏、SRAM、SDRAM。
· LTDC/DSI：RGB 屏、帧缓冲、同步时序、撕裂问题。

3.4 RTOS：FreeRTOS / RT-Thread
----------------------------------------------------------

FreeRTOS 是 15K 目标必须重点掌握的内容。

· 任务创建、任务删除、任务挂起、任务恢复。
· 优先级设计、时间片轮转、阻塞态、就绪态、运行态。
· 栈大小估算、栈溢出检测、High Water Mark。
· heap_1 到 heap_5，静态分配和动态分配。
· Queue：任务间消息传递、ISR 到任务传递。
· Binary Semaphore：中断同步。
· Counting Semaphore：资源计数。
· Mutex：共享资源保护、优先级继承、避免优先级反转。
· Event Group：多个事件条件组合。
· Direct-to-Task Notification：轻量通知，替代部分信号量。
· Software Timer：低频周期事件，不要在回调里做重活。
· Stream Buffer / Message Buffer：串口流、变长消息。
· CMSIS-RTOS v2：STM32CubeMX 常见封装。
· 调试重点：任务卡死、死锁、优先级设置错误、队列满、栈溢出、ISR API 用错。

3.5 通信协议
----------------

底层接口：

· UART 自定义帧：帧头、长度、命令、数据、CRC、超时、重发。
· RS485：方向脚控制、Modbus RTU、从机地址、功能码、CRC16。
· CAN：ID 分配、过滤器、周期报文、事件报文、心跳、错误恢复。
· I2C/SPI 传感器协议：寄存器读写、延时、复位、校准参数。

网络协议：

· lwIP 架构：link layer、IP、TCP/UDP、应用层。
· lwIP API：Raw API、Netconn API、Socket API。
· DHCP、静态 IP、DNS、ARP、ICMP、NTP/SNTP。
· TCP Server / TCP Client、UDP Server / UDP Client。
· HTTP Server：网页配置、REST 风格接口、JSON 参数。
· MQTT：连接、订阅、发布、QoS、KeepAlive、断线重连。
· TLS 基础认知：证书、内存占用、裁剪策略。MCU 上做 TLS 会明显增加 RAM/Flash 压力。

工业/物联网协议：

· Modbus RTU / TCP。
· 自定义二进制协议。
· CANopen 基本概念：对象字典、PDO、SDO、NMT。
· UDS / CCP / XCP 可作为汽车电子加分项，不是 STM32 15K 的第一优先级。
· LoRa、NB-IoT、4G Cat.1 常通过 AT 模块接入，重点是 AT 状态机和异常恢复。

3.6 屏幕驱动与 GUI
--------------------------

屏幕接口：

· OLED：I2C/SPI，SSD1306 类驱动。
· TFT SPI：ILI9341、ST7789、GC9A01 等。
· 8080 并口：FSMC/FMC 驱动，速度比 SPI 更高。
· RGB 屏：LTDC，使用帧缓冲，关注像素时钟和同步时序。
· MIPI DSI：更高阶，STM32H7/F7 部分型号涉及。

图形基础：

· 分辨率、像素格式、RGB565、ARGB8888、调色板。
· 点、线、矩形、图片、字体、抗锯齿。
· 帧缓冲、局部刷新、全屏刷新、双缓冲。
· DMA2D / Chrom-ART：填充、拷贝、颜色格式转换、图层混合。
· 触摸：电阻触摸、电容触摸、I2C触摸芯片、校准、去抖。
· LVGL：flush_cb、draw buffer、partial/direct/full render mode、input device、tick、task handler。
· TouchGFX：LTDC VSync、DMA2D、TouchController、Model-View-Presenter、资源生成。
· 常见问题：花屏、撕裂、刷新慢、触摸偏移、内存不够、DMA 与 Cache 冲突。

3.7 Bootloader / IAP / OTA
----------------------------------------------------

· Flash 分区：Bootloader、App、参数区、备份区、升级包区。
· App 跳转：关闭中断、设置 MSP、重定位向量表、跳转 Reset_Handler。
· 固件完整性：CRC32、长度、版本、魔术字。
· 固件安全：签名、加密、反回滚，至少了解。
· 升级方式：UART/YMODEM、USB、SD卡、HTTP、MQTT、私有TCP。
· A/B 双分区：新固件验证失败自动回滚。
· 断电保护：写入进度、状态机、升级标志、看门狗配合。
· 量产需求：出厂版本、设备序列号、配置保存、恢复出厂。

3.8 调试、测试与量产能力
----------------------------

· 调试工具：ST-Link、J-Link、OpenOCD、GDB、串口助手、逻辑分析仪、示波器。
· HardFault 分析：查看 CFSR、HFSR、BFAR、MMFAR、堆栈寄存器。
· 日志系统：等级、模块、时间戳、环形缓冲、掉电日志。
· 看门狗策略：任务喂狗表，不要简单在主循环无脑喂狗。
· 自动化测试：Python + pyserial、pytest、串口协议回归、CAN/Modbus脚本。
· 单元测试：Unity/Ceedling 或自建 host test。
· 静态分析：cppcheck、编译告警、格式化、复杂度。
· HIL：PC 控制串口/CAN/继电器/电源，自动验证板子行为。
· 量产工装：烧录、SN写入、校准、通信自检、传感器自检、测试报告导出。

3.9 硬件基础
----------------

· 能看懂原理图：电源、晶振、复位、下载口、外设连接。
· 上拉/下拉、开漏/推挽、电平转换。
· RS485 终端电阻、偏置电阻、隔离。
· CAN 终端电阻、收发器、总线错误。
· Ethernet PHY、RMII、晶振、变压器、PHY地址。
· LCD 背光、电源时序、FPC排线、触摸接口。
· 传感器信号：模拟量、数字量、滤波、校准。
· EMC/ESD 基础：TVS、磁珠、地线、线缆干扰。

【6. 高强度项目1：工业物联网控制器】
====================================

6.1 项目目标
----------------

做一个接近真实产品的 STM32 工业控制器：采集传感器，控制继电器/电机，通过 RS485/CAN 与现场设备通信，通过 Ethernet + lwIP + MQTT/HTTP 与上位机或云端通信，本地有日志、参数保存、看门狗和远程升级。

这个项目最贴近 15K-25K 岗位，因为它覆盖驱动、RTOS、协议栈、稳定性和文档。

6.2 推荐硬件
----------------

· MCU：STM32F407 + LAN8720，或 NUCLEO-F429ZI / NUCLEO-F746ZG / STM32H743。
· 通信：RS485模块、CAN收发器、以太网PHY或板载Ethernet。
· 传感器：SHT31/BME280、光照传感器、模拟电压输入。
· 执行器：继电器模块、PWM风扇或小电机。
· 存储：W25Qxx SPI Flash 或 SD卡。
· 调试：ST-Link、USB转串口、逻辑分析仪、示波器。

6.3 功能清单
----------------

· 传感器采集：温湿度、模拟量、电源电压。
· 执行器控制：继电器、PWM风扇、报警蜂鸣器。
· RS485 Modbus RTU：作为从站，支持读写寄存器。
· CAN：周期上报状态，接收控制命令。
· Ethernet + lwIP：DHCP/静态IP、TCP/UDP诊断。
· MQTT：周期上报数据，接收配置命令。
· HTTP Server：查看状态、修改参数、触发重启或升级。
· 参数保存：设备ID、网络参数、阈值、校准系数。
· 日志系统：运行日志、错误日志、复位原因。
· 看门狗：任务级喂狗，某任务卡死后复位并记录原因。
· OTA：HTTP 或 MQTT 分包升级，支持CRC校验。

6.4 RTOS任务设计
------------------------

任务  |  优先级  |  周期/触发  |  职责
———————————————————————————
sensor_task  |  中  |  500ms/1s  |  读取I2C/ADC数据，滤波，发布采集事件
control_task  |  中高  |  事件触发  |  根据阈值和远程命令控制继电器/PWM
modbus_task  |  中  |  串口IDLE触发  |  解析RS485帧，读写寄存器
can_task  |  中  |  周期+中断  |  发送心跳，处理CAN控制帧
net_task  |  高  |  lwIP线程/事件  |  维护网络状态，处理TCP/MQTT/HTTP
storage_task  |  低  |  队列触发  |  保存参数、写日志、Flash磨损控制
diag_task  |  低  |  CLI命令  |  输出系统状态、任务栈、网络状态
watchdog_task  |  高  |  200ms  |  检查各任务心跳，统一喂狗


6.5 软件模块
----------------

· bsp_gpio、bsp_uart、bsp_i2c、bsp_spi、bsp_can、bsp_eth。
· drv_sht31、drv_w25qxx、drv_rs485、drv_relay。
· app_sensor、app_control、app_protocol、app_network、app_ota。
· middleware_modbus、middleware_mqtt、middleware_http。
· sys_logger、sys_config、sys_watchdog、sys_fault。

6.6 必须考虑的异常
----------------------

· I2C传感器拔掉、SCL/SDA被拉低。
· RS485收到半包、粘包、CRC错误、地址不匹配。
· CAN总线关闭、过滤器配置错误、终端电阻缺失。
· 网线插拔、DHCP失败、MQTT Broker断开、DNS失败。
· Flash写入中断电，参数区损坏。
· 某个RTOS任务死循环或栈溢出。
· 升级包下载一半断网或断电。

6.7 验收标准
----------------

· 连续运行 72 小时不死机。
· MQTT断网重连 100 次后仍能恢复。
· Modbus错误帧压测 10000 帧不崩溃。
· 参数保存断电测试 100 次不丢配置。
· 看门狗复位后能记录是哪个任务超时。
· README 包含架构图、任务表、协议说明、测试报告。

6.8 简历写法
----------------

可写：

说明：基于 STM32F4/F7 + FreeRTOS + lwIP 实现工业物联网控制器，完成 I2C/ADC/CAN/RS485/Ethernet 驱动、Modbus RTU、MQTT、HTTP配置页、参数持久化、任务级看门狗和OTA升级；设计多任务消息架构，支持断网重连、异常帧恢复和72小时稳定性测试。

【7. 高强度项目2：联网HMI触摸屏控制面板】
============================================

7.1 项目目标
----------------

做一个 STM32 HMI 控制面板，重点突破屏幕驱动、LVGL/TouchGFX、触摸、DMA2D、双缓冲和联网配置。这个项目适合面向仪器仪表、医疗器械、工业控制屏、消费电子屏幕岗位。

7.2 推荐硬件
----------------

· STM32F429/F746/H743 开发板，优先选择带 LTDC + SDRAM + 触摸屏的板子。
· 低成本替代：STM32F407 + ILI9341 SPI屏 + XPT2046触摸。
· 显示：320x240、480x272、800x480 任选。

7.3 功能清单
----------------

· 启动画面、状态总览页、实时曲线页、参数设置页、日志页、网络页。
· 触摸输入：按钮、滑动、列表、弹窗、软键盘。
· 数据展示：温度/电压/网络状态/设备状态/告警。
· 参数修改：阈值、设备ID、IP、MQTT服务器。
· 本地日志：最近100条事件，可翻页。
· 网络联动：MQTT/HTTP收到新数据后UI刷新。
· 主题切换：日间/夜间。

7.4 驱动重点
----------------

· SPI屏：窗口设置、GRAM写入、DMA刷屏、局部刷新。
· LTDC屏：像素时钟、同步参数、前后肩、帧缓冲地址。
· DMA2D：矩形填充、内存拷贝、颜色转换。
· SDRAM：初始化时序、刷新周期、帧缓冲放置。
· 触摸：I2C/SPI读坐标、校准、去抖、长按/滑动。
· LVGL：flush_cb、buffer大小、tick、lv_timer_handler()、input driver。
· TouchGFX：VSync驱动、OSWrappers::signalVSync()、TouchController、资源生成。

7.5 性能指标
----------------

· UI切换无明显卡顿。
· 静态页面刷新 CPU 占用可说明。
· 局部刷新耗时可测量。
· 触摸延迟小于 100ms。
· 480x272 页面至少能做到 20 FPS 级别的流畅动画，具体取决于硬件和接口。
· SPI屏要接受刷新能力有限，重点展示局部刷新和DMA优化。

7.6 必须考虑的异常
----------------------

· 屏幕花屏、颜色反、方向错。
· 触摸坐标偏移、坐标轴反、抖动。
· 刷屏占用太高导致通信任务饿死。
· 字库和图片资源导致Flash/RAM不足。
· DMA2D和CPU同时访问帧缓冲引起撕裂。
· FreeRTOS任务优先级不当导致UI卡顿。

7.7 验收标准
----------------

· 有完整UI原型和真实板载运行效果。
· 支持至少3个页面和1个实时曲线。
· 能用诊断命令输出FPS、刷新耗时、内存占用。
· UI任务和通信任务互不阻塞。
· README 解释你为什么选择单缓冲/双缓冲/局部刷新。

【8. 高强度项目3：可靠Bootloader与OTA升级系统】
============================================================

8.1 项目目标
----------------

做一个可用于量产思路的 Bootloader：支持 UART 升级、网络升级、固件校验、版本管理、失败回滚、断电保护。这个项目面试非常好讲，因为它能体现你对产品可靠性的理解。

8.2 Flash分区示例
--------------------------

区域  |  内容  |  说明
————————————————
Bootloader  |  固定启动程序  |  只负责校验、升级、跳转
App A  |  当前稳定固件  |  默认运行
App B  |  新固件下载区  |  升级完成后验证
Param  |  参数区  |  设备ID、网络参数、版本
Log  |  升级日志  |  最近升级结果和错误码


8.3 功能清单
----------------

· 启动时检查App完整性。
· 支持串口命令进入升级模式。
· 支持 YMODEM 或自定义分包协议。
· 支持 HTTP/MQTT 下载固件。
· 固件头包含 magic、version、length、crc32、build time。
· 固件写入后校验CRC。
· 支持 A/B 分区切换。
· 新固件首次运行必须上报“启动成功”，否则下次回滚。
· 升级过程喂狗，异常不变砖。

8.4 关键难点
----------------

· 向量表跳转：MSP、Reset_Handler、VTOR。
· Flash擦写期间中断处理。
· App和Bootloader共享参数结构体版本。
· 升级状态机不能只靠一个flag，要区分 downloading、downloaded、verified、pending、confirmed、rollback。
· 断电场景必须逐步设计，不要一次性擦掉唯一可运行App。

8.5 验收标准
----------------

· 正常固件能升级成功。
· 错误CRC固件拒绝启动。
· 升级中断电可以恢复。
· 新固件启动失败能回滚旧版本。
· Bootloader和App版本信息可查询。
· 有升级协议文档和测试用例。

【12. 面试准备清单】

12.1 高频基础题
--------------------

· volatile 的作用是什么？中断变量为什么要加？
· 中断里能不能调用阻塞函数？为什么？
· DMA和CPU同时访问同一块内存会有什么问题？

· FreeRTOS中队列、信号量、互斥锁、事件组、任务通知分别适合什么场景？

· 栈溢出如何发现？
· 看门狗应该在哪个任务喂？
· lwIP Raw API、Netconn API、Socket API区别是什么？
· TCP和UDP区别？什么时候用UDP？
· MQTT KeepAlive和断线重连怎么做？
· Modbus RTU帧间隔为什么重要？
· CAN过滤器怎么配置？总线关闭怎么恢复？
· Bootloader如何跳转App？
· OTA升级断电如何不变砖？
· LCD双缓冲解决什么问题？
· LVGL的flush_cb作用是什么？

12.2 项目讲解模板
----------------------

面试讲项目时按这个顺序：

1. 项目背景：解决什么问题。
2. 硬件平台：MCU、外设、通信接口。
3. 软件架构：任务划分、模块划分、数据流。
4. 技术难点：比如断网重连、DMA丢包、屏幕撕裂、升级回滚。
5. 解决方案：具体到状态机、缓冲区、超时、日志、测试。
6. 结果数据：运行时长、丢包率、刷新耗时、升级成功率、测试次数。
7. 复盘：如果重做会优化什么。

· 能做基本屏幕驱动和 LVGL/TouchGFX 移植。
· 能做 Bootloader 或至少清楚 IAP 跳转和CRC校验。
· 能用示波器/逻辑分析仪定位通信问题。
· 能写清楚项目文档、协议文档和测试报告。

冲刺 20K+，你还需要：

· 长时间稳定性测试和自动化测试。
· 网络/屏幕/OTA至少一个方向能讲到深入细节。
· 有完整作品集，而不是零散Demo。
· 能从需求拆分架构、评估风险、设计异常恢复。

【15. 一句话路线】
==================

先把 STM32 外设和 C 语言打牢，再用 FreeRTOS 把驱动组织成产品架构，然后集中突破 lwIP联网、屏幕GUI、Bootloader/OTA 三个高薪模块，最后用一个完整项目和自动化测试把能力闭环。

【16. 整体知识关系图】
======================

STM32 嵌入式不是一堆互不相干的知识点，它更像一条从硬件信号到产品交付的链路：

[代码：text]
    硬件原理图
      -> MCU启动/时钟/中断
      -> GPIO/UART/I2C/SPI/CAN/ETH/LCD等外设
      -> DMA/中断/缓冲区/状态机
      -> 驱动层
      -> FreeRTOS任务、队列、事件、互斥
      -> 协议层：Modbus/CAN/MQTT/HTTP/TCP/UDP
      -> 应用层：采集、控制、显示、联网、升级、日志
      -> 稳定性：看门狗、异常恢复、HardFault、掉电保护
      -> 测试交付：自动化测试、文档、生产工装、面试作品集
[代码结束]

学的时候要始终问自己三个问题：

· 这个知识点解决什么真实问题？
· 它依赖前面哪个基础？
· 它会被后面哪个项目或岗位能力用到？

举例：

· UART 不是只会 HAL_UART_Transmit()。真正工作里会走到 UART + DMA + IDLE + 环形缓冲 + 帧解析 + Modbus/CLI + Python自动化测试。
· SPI 不是只会读一个寄存器。它会连接到 SPI Flash参数保存、TFT屏刷图、DMA刷屏、Bootloader读取升级包。
· FreeRTOS 不是只会创建任务。它连接到 通信任务不阻塞采集任务、屏幕刷新不能卡住网络、看门狗知道哪个任务死了。
· lwIP 不是只会 ping 通。它连接到 MAC/PHY硬件、DMA描述符、pbuf内存池、TCP长连接、MQTT断线重连、OTA下载。
· 屏幕驱动 不是只会显示图片。它连接到 帧缓冲内存、DMA2D、RTOS任务优先级、触摸输入、参数配置界面。
· Bootloader 不是只会跳转。它连接到 Flash分区、CRC、协议传输、掉电保护、版本管理、量产升级。

【17. 从零到15K的能力闭环】
==============================

17.1 第一个闭环：能控制硬件
--------------------------------

目标不是“点灯”，而是证明你理解 MCU 如何控制真实电路。

需要串起来：

· 原理图：LED 接在哪个引脚，按键是否上拉，继电器是否低电平触发。
· GPIO：输入、输出、复用、开漏、推挽。
· 时钟：外设时钟不开，寄存器或HAL配置都不会生效。
· 中断：按键触发、外部事件响应。
· 定时器：周期任务、PWM、测频、编码器。
· 调试工具：万用表确认电平，逻辑分析仪确认时序。

项目表现：

· 做一个按键控制继电器，同时串口输出状态。
· 做一个 PWM 调光或风扇调速，用示波器验证频率和占空比。
· 写 README 说明硬件连接、GPIO配置、测试结果。

面试表达：

说明：我不是只用 HAL 点灯，我会先看原理图确认引脚、电平、上拉下拉和复用功能，再配置时钟和GPIO模式。涉及周期控制时会使用定时器，涉及外部事件时会使用中断，并用逻辑分析仪或示波器验证实际波形。

17.2 第二个闭环：能稳定接收和解析数据
------------------------------------------

很多嵌入式问题本质上都是“数据不完整、不按预期、不稳定”。

需要串起来：

· UART/I2C/SPI/CAN/RS485：数据从哪里来。
· 中断/DMA：数据如何高效进入内存。
· 环形缓冲区：数据如何暂存，避免丢包。
· 状态机：如何处理半包、粘包、错误帧。
· CRC：如何判断数据是否可靠。
· 超时机制：数据缺失时如何退出等待。
· 日志：出现问题时如何定位。

项目表现：

· UART DMA + IDLE 接收任意长度数据。
· 支持半包、粘包、错误CRC、超时恢复。
· 写一个 Modbus RTU 从站或自定义二进制协议。
· 用 Python 连续发送随机帧做压力测试。

面试表达：

说明：串口接收我一般不会用阻塞轮询，而是用 DMA + IDLE 或中断接收，把数据放入环形缓冲区，再由协议任务做状态机解析。这样能处理粘包、半包、CRC错误和超时，ISR只做轻量通知，复杂解析放在任务里完成。

17.3 第三个闭环：能组织复杂任务
------------------------------------

单片机项目变复杂后，最大问题不是某个外设不会用，而是所有事情挤在一起。

需要串起来：

· 裸机 while：适合简单轮询。
· 定时器中断：适合精确周期触发，但不能做重活。
· FreeRTOS任务：适合拆分采集、通信、显示、存储、控制。
· 队列：任务之间传消息。
· 信号量：中断通知任务。
· 互斥锁：保护共享外设，如 I2C、SPI、Flash。
· 事件组：组合多个状态，如网络已连接、时间已同步、配置已加载。
· 软件定时器：低频周期事件。
· 看门狗：检查每个任务是否仍然健康。

项目表现：

· 采集任务每 1 秒读取传感器。
· 通信任务收到命令后修改参数。
· 存储任务异步保存配置。
· UI任务刷新屏幕，不阻塞网络任务。
· 看门狗任务检查所有关键任务心跳。

面试表达：

说明：我会先按功能和实时性拆任务。中断只负责搬运数据或释放信号量，业务逻辑放到任务里。任务间用队列传结构化消息，共享总线用互斥锁保护。看门狗不是随便喂，而是每个关键任务上报心跳，由watchdog任务统一判断。

17.4 第四个闭环：能联网并能恢复
------------------------------------

能 ping 通只是第一步，岗位真正看重的是网络异常后能不能恢复。

需要串起来：

· Ethernet硬件：MAC、PHY、RMII/MII、PHY地址、网线状态。
· DMA描述符：网卡收发数据靠DMA，内存和Cache要处理好。
· lwIP netif：把底层网卡接入协议栈。
· IP配置：DHCP、静态IP、DNS。
· TCP/UDP：传输层能力。
· HTTP/MQTT：应用层业务。
· 状态机：断网、重连、超时、心跳。
· 日志和诊断：能输出 IP、链路、连接、内存池、错误计数。

项目表现：

· DHCP获取IP，支持静态IP切换。
· MQTT周期上报数据，断网后自动重连。
· HTTP页面能查看状态和修改参数。
· 网线插拔 100 次、Broker 重启 100 次不死机。

面试表达：

说明：我把网络分成链路层、IP层、连接层和业务层来处理。网线断开时先更新链路状态，TCP/MQTT连接状态机进入重连流程，业务发送失败不会阻塞采集控制任务。诊断命令可以输出IP、PHY链路、TCP连接和lwIP内存池状态。

17.5 第五个闭环：能显示和交互
----------------------------------

屏幕方向的核心不是“做个页面”，而是显示性能、内存、触摸、任务协作。

需要串起来：

· 屏幕接口：SPI、8080、RGB/LTDC、DSI。
· 像素格式：RGB565、ARGB8888。
· 刷新策略：全刷、局部刷、单缓冲、双缓冲。
· DMA/DMA2D：降低CPU拷贝压力。
· SDRAM：大分辨率屏幕需要外部RAM。
· LVGL/TouchGFX：控件、页面、动画、输入设备。
· FreeRTOS：UI任务不能饿死通信任务。
· 触摸：坐标读取、校准、去抖。

项目表现：

· 做状态页、设置页、日志页、网络页。
· 实时曲线显示传感器数据。
· 触摸修改参数，保存到Flash。
· 网络收到数据后 UI 自动刷新。
· 统计 FPS、刷新耗时、内存占用。

面试表达：

说明：我做屏幕时会先确认接口带宽和内存预算。SPI屏通常用局部刷新和DMA，RGB屏会用LTDC帧缓冲，必要时用DMA2D做颜色转换和内存拷贝。GUI任务只处理显示事件，不直接阻塞等待通信或Flash写入。

17.6 第六个闭环：能升级和量产维护
--------------------------------------

产品卖出去后，最重要的问题之一是如何可靠升级。

需要串起来：

· Flash分区：Bootloader、App、参数区、备份区。
· 启动流程：Reset后先进入Bootloader，再决定跳App或升级。
· 向量表：设置 MSP、VTOR、跳转 Reset_Handler。
· 升级协议：UART/YMODEM、HTTP、MQTT分包。
· 完整性校验：CRC32、长度、版本、magic。
· 断电保护：状态机和A/B分区。
· 回滚：新固件不确认成功就恢复旧版本。
· 日志：记录升级失败原因。

项目表现：

· 串口升级和网络OTA至少实现一个。
· 支持错误固件拒绝启动。
· 支持升级中断电恢复。
· 支持版本查询和回滚。

面试表达：

说明：我设计Bootloader时会先规划Flash分区，Bootloader只负责校验、升级和跳转，不放复杂业务。升级包带magic、版本、长度和CRC。A/B分区可以避免擦掉唯一可运行固件，新固件首次启动后必须确认成功，否则下次启动回滚。

17.7 第七个闭环：能测试和证明稳定
--------------------------------------

15K以上岗位非常看重“你怎么证明它稳定”。

需要串起来：

· 日志：出现问题能追踪。
· 断言：开发阶段快速暴露错误。
· HardFault：能保存故障现场。
· 自动化脚本：重复测试协议。
· 压力测试：长时间、大数据、异常输入。
· HIL：硬件在环，模拟断电、断网、错误帧。
· 测试报告：把结果写清楚。

项目表现：

· Python脚本自动跑串口协议测试。
· MQTT断线重连压力测试。
· Modbus错误帧压力测试。
· Bootloader断电测试。
· 生成测试报告。

面试表达：

说明：我会给项目做基本自动化测试，比如用Python脚本通过串口或网络发送正常帧、错误帧、边界帧，统计成功率和响应时间。稳定性测试会覆盖长时间运行、断网重连、掉电恢复和看门狗复位记录。

【18. 核心知识点深讲】
======================

18.1 C语言为什么是根
--------------------------

STM32 的大多数高级问题，最后都会落回 C 语言和内存。

必须理解：

· 指针决定你能不能正确操作缓冲区、寄存器、DMA内存。
· 结构体决定你能不能设计协议帧、配置表、设备对象。
· 函数指针和回调决定你能不能写通用驱动和事件机制。
· volatile 决定中断和主程序共享变量是否会被编译器优化出问题。
· static 决定变量生命周期和模块封装。
· const 决定只读表、命令表、字体资源能否放到Flash。
· 内存对齐决定通信帧、DMA、结构体映射是否可靠。

和其他点的关系：

· UART环形缓冲需要指针和数组。
· Modbus寄存器表适合用结构体和表驱动。
· FreeRTOS任务参数本质是 void * 指针。
· lwIP 的 pbuf、netif、回调都离不开指针和结构体。
· LVGL/TouchGFX 里大量使用对象、回调、缓冲区。
· Bootloader固件头通常就是一个结构体。

建议练习：

· 写一个通用 ring_buffer，支持写入、读取、peek、丢弃、查找帧头。
· 写一个命令表，命令字符串映射到函数指针。
· 写一个协议帧结构，但不要直接把串口字节强转结构体，要考虑对齐和大小端。
· 写一个 CRC16/CRC32 模块，用于 Modbus 和固件校验。

18.2 中断、DMA、RTOS的边界
--------------------------------------

这是嵌入式工程里最容易混乱的地方。

中断适合做：

· 记录事件发生。
· 清标志位。
· 读取或提交少量数据。
· 释放信号量或任务通知。

中断不适合做：

· 长时间循环。
· 打印大量日志。
· Flash擦写。
· I2C/SPI阻塞通信。
· 动态内存分配。
· 复杂协议解析。

DMA适合做：

· UART连续接收。
· ADC多通道采样。
· SPI屏幕刷图。
· Ethernet收发包。
· SDIO/SDMMC数据搬运。

FreeRTOS任务适合做：

· 协议解析。
· 文件写入。
· 网络重连。
· UI刷新。
· 传感器周期采集。
· 业务状态机。

三者关系：

[代码：text]
    外设产生数据
      -> DMA搬运到内存
      -> 中断通知任务
      -> 任务解析数据
      -> 队列发送业务事件
      -> 应用任务处理事件
[代码结束]

典型例子：UART DMA + IDLE

1. UART收到字节，DMA自动写入缓冲区。
2. 发生IDLE中断，说明一段数据暂时结束。
3. ISR计算本次接收长度，放入环形缓冲或发送任务通知。
4. 协议任务从环形缓冲读取数据。
5. 状态机解析帧头、长度、命令、CRC。
6. 解析成功后投递到业务队列。

这个模型可以迁移到 CAN、Ethernet、SPI屏刷图、ADC采样。

18.3 HAL、LL、寄存器如何选择
--------------------------------------

实际工作里不用迷信某一种。

HAL适合：

· 快速搭建工程。
· 常规外设初始化。
· 团队成员水平不一致时保持可读性。
· CubeMX自动生成配置。

LL适合：

· 对性能和时序有要求的地方。
· HAL封装太重或不够灵活的地方。
· 精细控制寄存器位。

寄存器适合：

· 启动、时钟、中断、低功耗等底层问题定位。
· HAL/LL行为异常时查根因。
· 面试中证明你理解底层，而不是只会点库函数。

建议策略：

· 项目主体可以用 HAL 快速推进。
· 高频路径可用 LL 或寄存器优化，比如 GPIO快速翻转、定时器特殊模式。
· 所有 HAL 调用都要知道背后大致改了哪些寄存器。
· 学会读 Reference Manual，不要只看教程。

和岗位关系：

· 15K岗位通常接受 HAL，但希望你能解释中断、DMA、时钟、寄存器。
· 20K+岗位会更关注你是否能定位 HAL 封装背后的问题。

18.4 UART为什么要学到很深
----------------------------------

UART 是很多项目的调试口、协议口、模块口、生产口。

常见用途：

· 调试日志。
· CLI命令行。
· RS485/Modbus。
· 4G/NB-IoT/WiFi模块AT命令。
· Bootloader串口升级。
· 生产工装通信。

必须掌握：

· 阻塞发送和接收。
· 中断接收。
· DMA接收。
· IDLE空闲中断。
· 环形缓冲。
· printf重定向。
· 日志等级和异步输出。
· 帧解析和超时。
· 波特率误差。

和项目关系：

· 工业控制器：RS485 Modbus 和 CLI。
· Bootloader：串口升级。
· 自动化测试：Python通过串口跑回归。
· 模块联网：AT命令状态机。

常见坑：

· ISR里直接解析协议，导致中断占用过长。
· 接收缓冲区太小，数据覆盖。
· 没处理粘包和半包。
· printf阻塞导致实时任务卡顿。
· DMA接收后忘记处理Cache或缓冲区边界。
· RS485方向脚切换太早，最后一个字节还没发完。

18.5 I2C和SPI为什么常见但坑多
----------------------------------------

I2C特点：

· 两根线，多个设备，适合低速传感器。
· 依赖上拉电阻。
· 有ACK/NACK。
· 容易遇到总线挂死。

I2C必须掌握：

· 设备地址7位/8位区别。
· 寄存器读写流程。
· 多字节读写。
· 超时处理。
· SCL手动拉时钟恢复总线。
· 互斥锁保护共享I2C总线。

SPI特点：

· 速度快，适合 Flash、屏幕、ADC、无线模块。
· 需要片选。
· 有模式0/1/2/3。
· 可以配DMA。

SPI必须掌握：

· CPOL/CPHA。
· 片选时序。
· 全双工和半双工。
· DMA发送和接收。
· 多设备共享总线。
· 屏幕连续写GRAM。
· Flash页写入和扇区擦除。

二者与项目关系：

· I2C连接传感器和触摸芯片。
· SPI连接Flash和TFT屏。
· Flash用于参数保存、日志、升级包。
· 屏幕驱动依赖 SPI/FMC/LTDC，但SPI屏最适合练底层。

常见坑：

· I2C地址左移错误。
· I2C没有超时，设备异常后任务永久卡死。
· 多任务同时访问I2C导致数据错乱。
· SPI模式配置错误导致读到错误ID。
· SPI片选控制不规范导致多设备冲突。
· Flash写入前没擦除。

18.6 CAN、RS485和工业协议
--------------------------------------

工业现场最看重抗干扰、稳定、可诊断。

RS485适合：

· 中低速多节点。
· 布线较长。
· Modbus RTU。
· 传感器、仪表、PLC通信。

CAN适合：

· 实时性较好。
· 多主通信。
· 有仲裁机制。
· 汽车、机器人、电机控制、BMS。

Modbus RTU必须掌握：

· 从机地址。
· 功能码 03/04/06/16。
· 寄存器地址映射。
· CRC16。
· 帧间隔。
· 异常响应码。

CAN必须掌握：

· 标准帧和扩展帧。
· 数据帧和远程帧。
· 仲裁ID。
· 过滤器。
· 错误主动、错误被动、总线关闭。
· 终端电阻。
· 心跳和超时。

和项目关系：

· 工业控制器：Modbus RTU + CAN状态上报。
· 电机控制器：CAN接收目标速度/位置。
· 生产工装：通过RS485或CAN批量测试设备。

常见坑：

· RS485方向控制时序不对。
· 没有终端电阻或偏置电阻。
· Modbus寄存器地址和文档不一致。
· CAN波特率或采样点不匹配。
· CAN过滤器设置错误导致收不到包。
· 总线关闭后没有恢复策略。

18.7 lwIP要从硬件一路理解到应用
----------------------------------------

lwIP 学习不要从 MQTT 开始，应该从链路层开始。

层次关系：

[代码：text]
    RJ45/PHY
      -> RMII/MII
      -> STM32 MAC
      -> DMA Descriptor
      -> ethernetif.c
      -> lwIP netif
      -> IP/ARP/ICMP
      -> TCP/UDP
      -> HTTP/MQTT/SNTP
      -> 业务状态机
[代码结束]

必须掌握：

· PHY地址和链路状态。
· RMII时钟来源。
· MAC地址。
· DMA描述符和收发缓冲区。
· ethernetif_input() 或相关网卡输入流程。
· DHCP和静态IP。
· ARP和ping。
· TCP连接状态。
· UDP无连接特点。
· lwIP内存池配置。
· pbuf生命周期。
· Netconn/Socket/Raw API区别。

API选择：

· Raw API：性能好，但回调复杂，适合深入优化。
· Netconn API：和RTOS结合常见，结构较清晰。
· Socket API：最像PC编程，移植第三方库方便，但开销更大。

项目建议：

1. 先实现 DHCP + ping。
2. 再实现 UDP上报。
3. 再实现 TCP echo。
4. 再实现 HTTP配置页。
5. 最后实现 MQTT和OTA下载。

常见坑：

· H7/F7 Cache没处理，网络包随机异常。
· lwIP内存池太小，长时间运行后连接失败。
· TCP发送失败没处理重试。
· MQTT断开后状态机卡住。
· DHCP失败后没有静态IP兜底。
· 网线拔插后没有重新初始化链路状态。

面试亮点：

· 能说清楚 MAC/PHY、DMA、netif、pbuf、TCP连接状态。
· 能展示断网重连和长时间压测数据。
· 能提供诊断命令，例如 net status 输出 IP、link、memp、tcp state。

18.8 屏幕驱动要同时考虑带宽、内存和任务
--------------------------------------------

显示问题通常不是单纯“代码写错”，而是资源预算不对。

先算数据量：

· 320x240 RGB565 一帧约 150KB。
· 480x272 RGB565 一帧约 255KB。
· 800x480 RGB565 一帧约 750KB。
· 双缓冲直接乘以2。

接口影响：

· SPI屏：接口带宽有限，适合小屏和局部刷新。
· 8080并口/FMC：比SPI快，适合中等分辨率。
· RGB/LTDC：适合较大屏，但需要持续从帧缓冲取数据。
· DSI：更高阶，调试复杂。

内存影响：

· 小屏可以用内部 SRAM 做局部 buffer。
· 大屏通常要 SDRAM。
· LVGL draw buffer 不一定等于整屏帧缓冲。
· 图片、字体、动画会吃大量 Flash 和 RAM。

任务影响：

· UI刷新不能阻塞高优先级控制任务。
· 触摸输入要防抖和校准。
· 网络或传感器数据更新UI时，建议通过消息队列传事件。
· Flash写参数不能在UI回调里直接阻塞太久。

典型架构：

[代码：text]
    touch_irq / polling
      -> ui_task
      -> LVGL/TouchGFX
      -> display_flush
      -> DMA/SPI/LTDC/DMA2D
    
    sensor_task/network_task
      -> ui_event_queue
      -> ui_task刷新控件
[代码结束]

常见坑：

· SPI全屏刷新太慢，还想做复杂动画。
· LVGL buffer配置过大导致RAM不够。
· UI任务优先级太高，通信任务饿死。
· UI回调里直接写Flash或等待网络。
· LTDC时序配置不对导致花屏。
· DMA2D操作帧缓冲时没有处理同步。

面试亮点：

· 能说清楚为什么某块屏选择 SPI/FMC/LTDC。
· 能说明单缓冲、双缓冲、局部刷新的取舍。
· 能给出刷新耗时和内存占用数据。

18.9 Bootloader和OTA是产品意识的体现
------------------------------------------------------

Bootloader 涉及的知识非常综合：

· C语言结构体：固件头。
· Flash：擦写、分区、对齐。
· UART/Ethernet：升级包传输。
· CRC：完整性校验。
· 中断和向量表：App跳转。
· 看门狗：升级过程中不能卡死。
· 参数区：保存版本和升级状态。
· 测试：断电、错误包、回滚。

推荐状态机：

[代码：text]
    IDLE
      -> DOWNLOAD_REQUESTED
      -> DOWNLOADING
      -> DOWNLOADED
      -> VERIFYING
      -> VERIFIED
      -> PENDING_CONFIRM
      -> CONFIRMED
      -> ROLLBACK
[代码结束]

关键原则：

· Bootloader越简单越可靠。
· 不要擦掉唯一可运行App。
· 升级包未校验通过绝不跳转。
· 新App首次运行必须确认成功。
· 失败原因要记录，方便售后和现场定位。

常见坑：

· 跳转App前没有关闭中断。
· 没有设置MSP。
· VTOR没设置或地址不对。
· Flash擦写时被中断打断。
· 固件头和App链接地址不匹配。
· 升级状态只用一个flag，断电后状态混乱。

18.10 看门狗、HardFault和日志要一起设计
------------------------------------------------------

稳定性不是单个功能，而是一套系统。

看门狗：

· 不建议在主循环或某个定时器里无条件喂狗。
· 应该每个关键任务上报心跳。
· watchdog任务检查心跳、栈余量、队列状态。
· 如果某任务超时，记录任务ID和状态，再允许复位。

HardFault：

· 保存故障寄存器。
· 保存PC、LR、SP。
· 保存CFSR、HFSR、BFAR、MMFAR。
· 尽量把故障信息写到备份寄存器、Flash或保留RAM。
· 下次启动后输出故障原因。

日志：

· 分等级：ERROR、WARN、INFO、DEBUG。
· 分模块：NET、MODBUS、CAN、UI、OTA、STORAGE。
· 加时间戳。
· 支持环形缓冲。
· 支持导出最近日志。

三者关系：

[代码：text]
    任务异常
      -> 日志记录上下文
      -> 看门狗发现任务心跳超时
      -> 保存复位原因
      -> 系统复位
      -> 启动后读取故障信息
      -> 通过CLI/HTTP/屏幕展示
[代码结束]

面试亮点：

· “我不仅能让设备复位，还能知道为什么复位。”
· “我能把偶发问题转化为可定位的数据。”

【22. 学习时的先后关系】
========================

22.1 不建议跳跃学习
------------------------

不建议直接从屏幕、lwIP、OTA开始，因为这些都是综合应用。

正确顺序：

1. C语言、内存、模块化。
2. GPIO、时钟、中断、定时器。
3. UART/I2C/SPI/ADC/CAN 等外设。
4. DMA、环形缓冲、状态机。
5. FreeRTOS任务架构。
6. 工业协议和网络协议。
7. 屏幕GUI或lwIP专题突破。
8. Bootloader/OTA和稳定性。
9. 自动化测试和作品集整理。

22.2 为什么RTOS要放在外设之后
--------------------------------------

如果外设基础不牢，RTOS会把问题放大。

比如：

· UART裸机都收不稳，上RTOS后更难定位。
· I2C没有超时，上RTOS后可能把一个任务永久卡死。
· SPI片选时序不对，上RTOS后多任务共享总线会更混乱。
· 中断优先级不懂，FreeRTOS API在ISR中可能用错。

所以先把单个外设跑稳，再放进RTOS架构。

22.3 为什么lwIP要放在RTOS之后
------------------------------------------

lwIP通常涉及：

· 网络输入线程。
· TCP/IP线程。
· 应用任务。
· 内存池。
· 超时定时器。
· 连接状态机。

如果RTOS任务、队列、互斥、栈、堆不熟，lwIP问题会很难排。

22.4 为什么屏幕GUI要和内存一起学
----------------------------------------

屏幕不是简单外设，它直接挑战内存和带宽。

如果不理解 RAM、DMA、缓存、刷新区域，就很容易出现：

· RAM不够。
· 刷新慢。
· 花屏。
· 撕裂。
· UI卡顿。
· 通信任务被拖慢。

22.5 为什么OTA要放在后期
--------------------------------

OTA需要前面很多能力：

· Flash擦写。
· 串口或网络传输。
· CRC校验。
· Bootloader跳转。
· 状态机。
· 掉电测试。
· 版本管理。

它不是孤立模块，是产品成熟度的综合体现。

【24. 15K简历项目应该写到什么粒度】
======================================

不要写：

说明：熟悉 STM32，熟悉 FreeRTOS，熟悉 lwIP。

这样太空。

应该写：

说明：基于 STM32F407 + FreeRTOS + lwIP 实现工业物联网控制器，负责 UART DMA + IDLE 串口协议、RS485 Modbus RTU、CAN心跳、I2C传感器采集、W25Qxx参数保存、MQTT断线重连、HTTP配置页和任务级看门狗。设计采集、控制、通信、存储、网络、诊断多任务架构，通过队列传递业务事件，通过互斥锁保护共享I2C/SPI总线。完成72小时稳定性测试、Modbus错误帧压测和MQTT断网重连测试。

再写技术难点：

· 使用 UART DMA + IDLE + 环形缓冲解决串口半包、粘包和高频数据丢包问题。
· 使用 FreeRTOS 队列和事件组解耦采集、控制、网络任务，避免网络阻塞影响本地控制。
· 为 MQTT 设计连接状态机，支持 DNS失败、Broker断开、网线插拔后的自动恢复。
· 设计 Flash 参数双备份和CRC校验，避免掉电导致配置损坏。
· 设计任务级看门狗和复位原因记录，用于定位偶发卡死。

这样面试官会认为你真的做过完整项目。

【25. 面试时如何把点讲成线】
============================

25.1 UART问题的完整回答方式
------------------------------------

面试官问：你怎么做串口接收？

普通回答：

说明：我用HAL库串口中断接收。

更好的回答：

说明：如果数据量小，可以用中断接收；如果数据长度不固定或数据量较大，我会用 DMA + IDLE。DMA负责搬运数据，IDLE中断说明一段数据结束。中断里只计算长度并通知协议任务，协议任务从环形缓冲区取数据，用状态机处理帧头、长度、命令、CRC和超时。这样可以处理半包、粘包和错误帧，也不会在中断里做复杂解析。

25.2 FreeRTOS问题的完整回答方式
--------------------------------------------

面试官问：你怎么划分任务？

更好的回答：

说明：我会先按实时性和阻塞风险划分。比如采集任务周期读取传感器，控制任务处理本地保护，通信任务处理串口/CAN，网络任务处理MQTT/HTTP，存储任务异步写Flash，诊断任务处理CLI。任务之间用队列传事件，共享I2C/SPI用互斥锁保护。中断只释放信号量或任务通知，不直接做业务逻辑。最后用任务级看门狗检查每个关键任务心跳。

25.3 lwIP问题的完整回答方式
------------------------------------

面试官问：你做过lwIP吗？

更好的回答：

说明：做过基础以太网接入和MQTT/HTTP应用。我会先确认PHY地址、RMII时钟和链路状态，再把底层驱动接入lwIP netif。应用层用状态机维护DHCP、TCP连接和MQTT连接。断网时业务发送不会一直阻塞，连接恢复后重新订阅和上报。调试时会输出IP、link状态、TCP状态和lwIP内存池信息。F7/H7平台还需要特别注意D-Cache和DMA缓冲区一致性。

25.4 屏幕问题的完整回答方式
--------------------------------

面试官问：你做过屏幕吗？

更好的回答：

说明：做过SPI TFT或LTDC屏幕移植。屏幕我会先算分辨率、像素格式和帧缓冲大小，再根据接口带宽决定全刷还是局部刷新。SPI屏通常用DMA和局部刷新，RGB屏用LTDC帧缓冲，复杂拷贝或颜色转换可以用DMA2D。LVGL移植时重点是flush_cb、draw buffer、tick和触摸输入。UI任务通过队列接收传感器或网络事件，不直接阻塞等待外设。

25.5 OTA问题的完整回答方式
----------------------------------

面试官问：OTA怎么保证不变砖？

更好的回答：

说明：关键是Flash分区和升级状态机。Bootloader固定不动，App分A/B区或至少有备份区。新固件下载到非运行区，固件头包含magic、版本、长度和CRC，校验通过后才标记为待启动。新App首次启动后必须写确认标志，如果没有确认成功，下次Bootloader回滚旧版本。升级状态要区分下载中、已下载、校验通过、待确认、已确认、回滚，不能只用一个flag。

【26. 最终冲刺建议】
====================

如果你要最快贴近 15K 岗位，按这个优先级投入：

1. UART DMA + IDLE + 环形缓冲 + 协议状态机
2. FreeRTOS任务架构 + 队列/互斥/事件组 + 任务级看门狗
3. RS485 Modbus RTU + CAN
4. I2C/SPI传感器和Flash驱动
5. lwIP DHCP/TCP/UDP/MQTT/HTTP + 断网重连
6. LVGL或TouchGFX屏幕项目
7. Bootloader/OTA + Flash分区 + CRC + 回滚
8. Python自动化测试 + 测试报告

最终作品集建议只有三个，但每个都做深：

· 工业物联网控制器：主项目，覆盖最多岗位要求。
· HMI触摸屏控制面板：证明屏幕和GUI能力。
· Bootloader/OTA升级系统：证明产品可靠性和量产意识。

你不需要把每个方向都做到专家级，但至少要有一个方向能讲得很深。对 15K 左右 STM32 岗位，最推荐深挖 FreeRTOS + lwIP + 工业协议 + 稳定性测试 这条线，因为它最贴近真实产品交付。
